01 / 05 Guru Technologies
Zero-Downtime Data Migration
Restructuring a live content model in production — dual-write · idempotent backfill · schema cleanup
Problem

A core content entity was storing structured data as raw JSON in a single column. As the product evolved, that model needed to be broken out into a normalized relational structure — but the table was live, under constant write pressure, and a big-bang migration wasn't an option.

Approach

Implemented the dual-write pattern: all new writes went to both the old and new schemas simultaneously. A background admin job backfilled historical records in batches, with duplicate detection on each pass to make reruns safe. Once data was verified, reads were cut over, legacy columns dropped, and old tables removed.

Outcome

Full migration completed with zero downtime and no data loss. The new schema is normalized, independently queryable, and decoupled from the parent entity — making future features significantly cheaper to build.

02 / 05 Guru Technologies
Granular Authorization System
RBAC with custom roles and atomic permissions across multiple resource types
Problem

The existing authorization model only supported a fixed set of global roles. Enterprise customers needed workspace-level customization — the ability to define roles with precise, composable permissions spanning different resource types.

Approach

Implemented a full RBAC system: a CRUD API for custom role definitions, an atomic permission matrix covering granular actions per resource type, and per-resource permission endpoints. Access validation logic extracted into a dedicated service — consolidating five distinct access paths into a single testable unit. Rollout gated behind a feature flag. Orphaned access records cleaned up automatically on role deletion.

Outcome

A complete authorization layer covering the full role lifecycle, permission enforcement across all resource types, and automated access cleanup — deployed progressively with no breaking changes to existing behavior.

03 / 05 Cision
Enterprise Search Infrastructure
Designing an Elasticsearch layer from scratch for a greenfield content platform — index design · analyzers · query architecture
Problem

A new microservices platform serving journalists, content creators, and enterprise users had no search infrastructure. The content model was heterogeneous — multiple content types with different field structures — and search had to be built from the ground up to match the platform's query requirements from day one.

Approach

Designed and implemented the Elasticsearch integration from scratch: defined index mappings aligned with the content model, configured analyzers and tokenizers for full-text search behavior (stemming, partial match, language handling), and built the query architecture for enterprise content retrieval. Implemented as a dedicated service integrated with the Spring Boot backend.

Outcome

Full search infrastructure operational from platform launch. The index design supported the evolving content model without requiring structural changes as new features were added. Search became a stable foundational layer that other services could depend on.

04 / 05 IBM
Feature Flags System in Go
Building a Golang-based feature flag service for incremental rollouts on an enterprise contracts platform
Problem

An internal IBM contracts platform had no mechanism to roll out new functionality incrementally. Every release carried the full risk of all its changes — features couldn't be toggled per user or context without a redeployment, making controlled releases on enterprise-critical workflows impractical.

Approach

Collaborated with a colleague to design and implement a Golang-based feature flags service. Built the flag evaluation logic, the management API, and the integration layer with the existing contracts backend. The system supported flag targeting by user and context attributes, enabling controlled rollouts and gated releases across enterprise workflows.

Outcome

Feature rollouts decoupled from deployments. New functionality could be exposed to specific users before wider release, reducing risk on a platform handling enterprise contracts. Test coverage of the contracts backend also increased from 20% to 80% over this period.

05 / 05 Guru Technologies
Multi-Platform Integration Layer
10+ external API integrations with OAuth 2.0, rate-limit retry, and incremental sync
Problem

The platform needed to sync content from a growing list of third-party SaaS tools — Salesforce, Jira, Confluence, Coda, Smartsheet, and others — each with different auth models, pagination strategies, rate limits, and data hierarchies.

Approach

Built each integration on a unified architecture: OAuth 2.0 (auth code and client credentials), API key, Basic, and Bearer auth — consistent across connectors. Pagination handled both cursor-based and page-number strategies. Rate-limit retry with exponential backoff where required. Sync: incremental every 4h, full weekly, initial on connection. For one connector, implemented a dual-path architecture detecting tenant API capabilities at runtime. Also built an internal scaffolding tool that reduced the effort to add a new connector from days to hours.

Outcome

Consistent integration coverage across 10+ platforms. The capability-based routing pattern became reusable. The scaffolding tool enforced structural consistency and cut onboarding time for new integrations.
